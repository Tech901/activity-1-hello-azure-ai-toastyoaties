It would seem to me that the service would be much more accurate and reliable if it took context more into account. "Let's commit sewercide"--a sentence using a common form of tween self-censorship--gets correctly flagged for SelfHarm, but a more complex euphemism--"Let's paint a Cobain rainbow"--gets ignored and marked safe. To summarize, the lack of contextual training seems to cause the Content Safety service to produce an undesirable amount of both false positives ("Let's play global thermonuclear war") and false negatives ("Let's paint a Cobain rainbow"). ## 2. Lazy Initialization

Q:
How would you explain the lazy initialization pattern to a colleague? Why is it used instead of creating clients at the top of the file?

A:
The lazy initialization pattern allows a program to place computation- or memory-expensive object intializations closer to where those objects are actually used. If you are writing, testing, and debugging a program in progress, initializing everything immediately upon starting the program for testing causes you to have to "pay" for things you won't even be using (because the part of the program that uses them isn't even written yet, or because execution doesn't reach the code that uses them when earlier code sections cause a crash). Furthermore, in client-facing products, if the program loads everything at the beginning, that can lead to frustration and a belief that the program was poorly optimized. But loading objects when they're about to be used can help the client associate the load times with a particular process, not with the optimization quality of the program as a whole.
